# SPIRE: SPIFFE Testing Server & Why pid: host Is Required

**Understanding the local SPIRE test harness, why it's structured the way it is, and why cross-container Workload API calls need `pid: "host"`**

---

## What is SPIRE?

SPIRE (the SPIFFE Runtime Environment) is the reference implementation of SPIFFE — it issues **SVIDs** (SPIFFE Verifiable Identity Documents) to workloads after **attestation**, not after a domain-ownership challenge like ACME's HTTP-01/DNS-01.

```
ACME (Pebble)
    ↓
    [Subject: domain name]
    [Auth: HTTP-01/DNS-01 challenge — proves control of a domain]
    [Trust root: public CA / public WebPKI]

SPIFFE (SPIRE)
    ↓
    [Subject: SPIFFE ID — a workload's identity, not a domain]
    [Auth: node + workload attestation — proves what/where a process is]
    [Trust root: your own SPIRE server / trust domain, never the public WebPKI]
```

## Why This Harness Is Two Phases, Not One

Pebble's harness is a single `docker compose up -d` because Pebble needs no bootstrap step. SPIRE does: the agent authenticates to the server using a **join token**, and that token has to be generated by the server *before* the agent can start. There's no way around this — it's the same bootstrap SPIRE requires in any real deployment (just usually automated via cloud/k8s node attestation instead of a manual token).

```bash
# Phase 1 — bring up the server, generate a join token + register a demo workload
docker compose -f docker-compose.spire.yml up -d spire-server
./spire/register-workload.sh

# Phase 2 — start the agent (and the test runner) with that token
SPIRE_JOIN_TOKEN=$(cat spire/data/join_token.txt) \
  docker compose -f docker-compose.spire.yml up -d spire-agent spire-test
```

`register-workload.sh` runs on the **host**, not inside a container — the official `spire-server`/`spire-agent` images are minimal/distroless (no shell), so registration uses `docker compose exec spire-server /opt/spire/bin/spire-server ...` directly rather than a script executed inside the container.

## The pid: "host" Problem

### What breaks without it

Bring the harness up without `pid: "host"` and a fetch from a separate container fails:

```
rpc error: code = Unavailable desc = connection error: desc = "error reading server preface: read unix @->/tmp/spire-agent/public/api.sock: read: connection reset by peer"
```

The agent's own logs show the real cause:

```
level=warning msg="Connection failed during accept" error="could not resolve caller information" subsystem_name=endpoints
```

### Why it happens

SPIRE's `unix` WorkloadAttestor identifies a connecting workload by resolving the caller's UID (and other process facts) via `/proc/<pid>` **as seen from the SPIRE Agent container's own PID namespace**. Docker gives every container its own PID namespace by default — so when a workload in a *different* container connects to the agent's Workload API socket, the agent can see the raw UID over `SO_PEERCRED` but can't resolve `/proc/<pid>` for a PID that doesn't exist in its namespace, and attestation fails.

This has a real production analogue: it's exactly why the SPIRE Agent DaemonSet in Kubernetes requires `hostPID: true`. `pid: "host"` in `docker-compose.spire.yml` is the local-Compose equivalent — both `spire-agent` and any workload container calling its Workload API (here, `spire-test`) need it.

### What still works without it

Node attestation (agent ↔ server) doesn't need this — it's a gRPC call over the network (`spire-server:8081`), not a Unix socket peer-credential lookup. Only the **Workload API** (agent ↔ workload, over the Unix socket) needs matching PID namespaces.

---

## Why There's No ACME_INSECURE Equivalent

Pebble needs `ACME_INSECURE=true` because it serves HTTPS with a self-signed certificate that isn't in any trust store. SPIRE has no equivalent flag, because there's nothing analogous to disable:

- The Workload API is a **local Unix domain socket**, not a network TLS endpoint — there's no certificate to skip-verify.
- Node attestation between agent and server does use TLS, but via SPIRE's own bootstrap trust (the `insecure_bootstrap = true` setting in `spire/agent/agent.conf`, dev-only, skips out-of-band bootstrap bundle verification — the closest analogue, already set for this harness).
- Nothing here ever touches the public WebPKI, so there's no "self-signed cert rejected by the system trust store" failure mode to work around in the first place.

## How to Use the Harness

### Start it

```bash
# Phase 1: server + registration
docker compose -f docker-compose.spire.yml up -d spire-server
./spire/register-workload.sh

# Phase 2: agent + test runner, using the token phase 1 generated
SPIRE_JOIN_TOKEN=$(cat spire/data/join_token.txt) \
  docker compose -f docker-compose.spire.yml up -d spire-agent spire-test
```

### Register an additional workload

Registration entries map a **selector** (how the agent recognizes a workload — e.g. `unix:uid:1000`) to a **SPIFFE ID**. Add more via:

```bash
docker compose -f docker-compose.spire.yml exec spire-server \
  /opt/spire/bin/spire-server entry create \
  -spiffeID spiffe://example.org/workload/my-new-workload \
  -parentID spiffe://example.org/agent \
  -selector unix:uid:1000
```

### Smoke-test a fetch manually

```bash
docker run --rm --pid host \
  -v spiffe-svid-issuance_spire-agent-socket:/tmp/spire-agent/public \
  --entrypoint /opt/spire/bin/spire-agent ghcr.io/spiffe/spire-agent:1.9.6 \
  api fetch x509 -socketPath /tmp/spire-agent/public/api.sock
```

An unregistered UID gets a clean, testable rejection rather than a crash:

```
rpc error: code = PermissionDenied desc = no identity issued
```

### Stop it

```bash
docker compose -f docker-compose.spire.yml down -v
```

`-v` also drops the SQLite-backed server data volume — registration entries and the CA don't survive without it. Omit `-v` if you want entries to persist across restarts (the server datastore is a named volume, so `register-workload.sh` is safe to re-run — it treats "entry already exists" as a no-op, not an error).

## Common Issues

### Issue: "could not resolve caller information" in spire-agent logs

**Cause:** `pid: "host"` missing from either `spire-agent` or the calling container.

**Fix:** Confirm both services have `pid: "host"` in `docker-compose.spire.yml`.

### Issue: `SPIRE_JOIN_TOKEN not set` / agent won't start

**Cause:** Trying to bring up `spire-agent` before running `register-workload.sh`, or forgetting to export the token.

**Fix:**
```bash
SPIRE_JOIN_TOKEN=$(cat spire/data/join_token.txt) \
  docker compose -f docker-compose.spire.yml up -d spire-agent
```

### Issue: `PermissionDenied: no identity issued` for a workload you expect to be registered

**Cause:** No registration entry matches that workload's selectors (commonly a UID mismatch — check what UID the calling container actually runs as with `id` inside it).

**Fix:** Register an entry with the correct selector (see "Register an additional workload" above).

## Summary

| Aspect | Pebble (ACME) | SPIRE (SPIFFE) |
|--------|---------------|----------------|
| **Subject** | Domain name | SPIFFE ID (workload identity) |
| **Auth mechanism** | HTTP-01/DNS-01 challenge | Node + workload attestation |
| **Trust root** | Public WebPKI (bypassed via `ACME_INSECURE` for this self-signed test server) | Your own SPIRE server (never public WebPKI, by design) |
| **Transport** | HTTPS (network) | Unix domain socket (local only) |
| **Startup** | Single `docker compose up -d` | Two-phase: generate join token, then start agent |
| **Container isolation gotcha** | None | Needs `pid: "host"` for cross-container Workload API calls |
| **Unregistered subject behavior** | N/A (ACME doesn't pre-register domains) | Clean `PermissionDenied: no identity issued` |

**Key takeaway:** SPIRE's local test harness needs a join-token bootstrap step and shared PID namespaces that Pebble never required, because SPIFFE's entire authentication model — attestation rooted in your own infrastructure — is fundamentally different from ACME's domain-challenge-against-a-public-CA model. This is not a testing shortcut to later "turn off" for production, unlike `ACME_INSECURE`; it's the real SPIFFE trust model, working exactly as designed even in this local harness.
